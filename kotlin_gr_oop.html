Αντικειμενοστραφής προγραμματισμός

Τα αρχικά OOP σημαίνουν Object-Oriented Programming (Αντικειμενοστραφής προγραμματισμός).

Ο διαδικαστικός προγραμματισμός έχει να κάνει με το γράψιμο διαδικασιών ή μεθόδων που εκτελούν εργασίες πάνω στα δεδομένα, ενώ ο αντικειμενοστραφής προγραμματισμός έχει να κάνει με τη δημιουργία αντικειμένων που περιέχουν και δεδομένα και μεθόδους.

Ο αντικειμενοστραφής προγραμματισμός έχει πολλά πλεονεκτήματα σε σχέση με τον αντικειμενοστραφή προγραμματισμό:

  • Το OOP πρόγραμμα είναι ταχύτερο και ευκολότερο να εκτελεστεί
  • Το OOP παρέχει ξεκάθαρη δομή των προγραμμάτων
  • ΤΟ OOP βοηθά στη διατήρηση του κώδικα της kotlin να παραμείνει DRY ("Don't Repeat Yourself"), και κάνει τον κώδικα ευκολότερο να συντηρηθεί να τροποποιηθεί και να αποσφαλματωθεί.
  • Το OOP κάνει εφιτκή τη δημιουργία πλήρως επαναχρησιμοποιήσιμων εφαρμογών με λιγότερο κώδικα και μικρότερο χρόνο ανάπτυξής του

Σχόλιο: Η αρχή "Don't Repeat Yourself" (DRY) έχει να κάνει με τη μείωση του επαναλήψιμου κώδικα. Πρέπει να αφαιρέσουμε τα κομμάτια κώδικα που είναι κοινά σε μία εφαρμογή και να τα τοποθετήσουμε σε ένα μόνο μέρος προκειμένου να τα επαναχρησιμοποιούμε αντί να τα επαναλαμβάνουμε.

Επομένως, η κλάση είναι ένα πρότυπο μοντελοποίησης αντικειμένων και το αντικείμενο είναι ένα στιγμιότυπο (instance) μίας κλάσης.

Όταν δημιουργιούνται μεμονωμένα αντικείμενα, κληρονομούν όλες τις μεταβλητές και τις μεθόδους από την κλάση.

Packages και imports

Ένα source file μπορεί να ξεκινά με τη δήλωση πακέτου (package declaration): 

package  org.example
    fun  printMessage() { /*...*/ }
    class Message {  /*...*/ }

    // ...

Όλα τα περιεχόμενα - όπως οι κλάσεις και οι συναρτήσεις - του source file περιλαμβάνονται σε αυτό το πακέτο. Επομένως, στο παραπάνω παράδειγμα το πλήρες όνομα της συνάρτησης printMessage() είναι org.example.printMessage και το πλήρες όνομα της κλάσης Message είναι org.example.Message.

Αν το πακέτο δεν είναι προσδιορισμένο, τα περιεχόμενα ενός τέτοιου αρχείου ανήκουν στο default πακέτο χωρίς όνομα.

Default imports

Ένας αριθμός πακέτων από σύμβαση είναι εισαγμένο (imported) σε κάθε αρχείο Kotlin:

  • kotlin.*
  • kotlin.annotation.*
  • kotlin.collections.*
  • kotlin.comparisons.*
  • kotlin.io.*
  • kotlin.ranges.*
  • kotlin.sequences.*
  • kotlin.text.*

Επιπρόσθετα πακέτα εισάγονται ανάλογα με την στοχευμένη πλατφόρμα:

  • JVM:
    • java.lang.*
    • kotlin.jvm.*
  • JS:
    • kotlin.js.*

Imports

Ξεχωριστά από τα default imports, κάθε αρχείο περιέχει τις δικές του εισηγμένες οδηγίες.

Μπορούμε να εισάγουμε είτε ένα μόνο όνομα: 

import org.example.Message // Η κλάση Message είναι τώρα προσβάσιμη δίχως προϋποθέσεις

ή όλα τα προσβάσιμα περιεχόμενα (ενός πακέτου, μία κλάσης, ενός αντικειμένου κοκ): 

import org.example.* // Όλα όσα περιέχονται στο πακέτο 'org.example' γίνονται προσβάσιμα

Αν υπάρχει διαμάχη ονομάτων, μπορούμε να το αποσαφηνίσουμε χρησιμοποιώντας ένα keyword για να μετονομάσουμε τοπικά την υπό διαμάχη οντότητα: 

import org.example.Message // Η κλάση Message είναι προσβάσιμη
import org.test.Message as TestMessage // Το TestMessage αντικαθιστά το 'org.test.Message'

Το keyword import δεν είναι περιορισμένο στην εισαγωγή κλάσεων· μπορούμε επίσης να το χρησιμοποιήσουμε για να εισάγουμε άλλες δηλώσεις:

  • Top-level συναρτήσεις και ιδιότητες
  • Συναρτήσεις και ιδιότητες που δηλώνονται σε δηλώσεις αντικειμένων (object declarations)
  • Enum constants

Ορατότητα των top-level δηλώσεων

Αν μία top-level δήλωση έχει σημειωθεί ως private, είναι private στο αρχείο που έχει δηλωθεί (βλέπε Visibility modifiers).

Interfaces

Τα interfaces στην Kotlin μπορούν να περιλαμβάνουν δηλώσεις μεθόδων, καθώς επίσης και υλοποιήσεις μεθόδων. Αυτό που κάνει τη διαφορά από τις abstract κλάσεις έγκειται στο γεγονός πως τα interfaces δεν μπορούν να αποθηκεύουν καταστάσεις. Μπορούν να έχουν ιδιότητες, όμως αυτές χρειάζεται να είναι abstract ή να παρέχουν υλοποιήσεις πρόσβασης (accessor implementations).

Ένα interface ορίζεται χρησιμοποιώντας το keyword interface: 

interface MyInterface {
    fun  bar()
    fun  foo() {
    // προαιρετικά σώμα του interface
    }
}

Υλοποίηση interfaces

Μία κλάση ή ένα αντικέιμενο μπορεί να υλοποιήσει ένα ή περισσότερα interfaces: 

class Child : MyInterface {
    override fun  bar()
        // σώμα
    }
}

Ιδιότητες (Properties) στα interfaces

Μπορούμε να δηλώσουμε ιδιότητες στα interfaces. Μία δηλωμένη ιδιότητα σε κάποιο interface μπορεί να είναι είτε abstract ή να παρέχει υλοποιήσεις πρόσβασης (implementations for accessors). Οι δηλωμένες ιδιότητες στα interfaces δεν μπορούν να έχουν υποστηρικτικά πεδία, και για αυτό οι δηλωμένοι accessors στα interfaces δεν μπορούν να αναφερθούν σε αυτές: 

interface MyInterface {
    val prop: Int  // abstract
                
    val propertyWithImplementation: String
    get() = "foo"

    fun foo() {
        print(prop)
    }
}

class Child : MyInterface {
     override val prop: Int = 29
}

Κληρονομικότητα στα interfaces

Ένα interface μπορεί να κληρονομήσει από άλλα interfaces, εννοώντας πως μπορεί να παρέχει υλοποιήσεις για τα μέλη του και να δηλώνει νέες συναρτήσεις και ιδιότητες. Φυσικά, οι κλάσεις που υλοποιούν ένα τέτοιο interface απαιτείται μόνο να ορίζουν τις υλοποιήσεις που λείπουν: 

interface Named {
    val name: String 
}                        

interface Person : Named {
    val firstName: String
    val lastName: String
            
    override val name: String get() = "$firstName $lastName"
}

data class Employee(
    // Η υλοποίηση της μεταβλητής 'name' δεν απαιτείται
    override val firstName: String,
    override val lastName: String,
    val position: Position
) : Person

Επίλυση διαμάχης σε περίπτωση υπερκάλυψης (Resolving overriding conflicts)

Όταν δηλώνουμε πολλούς τύπους στην supertype list, μπορούμε να κληρονομήσουμε περισσότερες από μία υλοποιήσεις της ίδιας μεθόδου: 

interface A {
    fun foo() { print("A") }
    fun bar()
}                        
            
interface B {
    fun foo() { print("B") }
    fun bar() { print("bar") }
}

class C : A {
    override fun bar() { print("bar"}   
}

class D : A, B {
    override fun foo()) { 
        super<A>.foo()
        super<B>.foo()   
}

    override fun bar() {
        super<B>.bar()   
    }   
}

Τα interfaces A και B δηλώνουν και τα δύο τις συναρτήσεις foo() και bar(). Και τα δύο υλοποιούν τη foo(), όμως μόνον η B υλοποιεί τη bar(). Η συνάρτηση bar() δεν είναι σημειωμένη ως abstract στο A, επειδή αυτός είναι από σύμβαση ο τρόπος για τα interfaces αν η συνάρτηση δεν έχει σώμα). Τώρα, αν μία concrete κλάση C παράγεται απο το A, χρειάζεται να κάνουμε override την συνάρτηση bar() και να παρέχουμε την υλοποίηση.

Ωστόσο, αν η κλάση D προέρχεται από τα A και B, χρειάζεται να υλοποιήσουμε όλες τις μεθόδους που έχουμε κληρονομήσει από τα διάφορα interfaces, και χρειάζεται να προσδιορίσουμε πώς ακριβώς το D θα πρέπει να τις υλοποιήσει. Αυτός ο κανόνας ισχύει και για τις μεθόδους για τις οποίες έχουμε κληρονομήσει μία υλοποίηση (bar()) και για αυτές που έχουμε κληρονομήσει πολλαπλές (foo() ).
